## Migration Support

When breaking changes are necessary:

1. **Deprecation Notice**: Mark as deprecated in the previous minor version
2. **Migration Guide**: Provide step-by-step migration instructions
3. **Codemods**: Automated migration scripts when possible
4. **Transition Period**: Keep deprecated version available for at least one minor release

## Release Cadence

- **Patch releases**: As needed for bug fixes
- **Minor releases**: Monthly (or as features are ready)
- **Major releases**: Annually (or when necessary for significant changes)

### Tag Convention

- **Format**: `vX.Y.Z` or `vX.Y.Z-alpha/beta`
- **Examples**: `v0.7.0-alpha`, `v1.0.0`
- Tags trigger npm publish via CI/CD

### Version Synchronization

**The Changelog is the source of truth.**

Before creating a release:

1. Add entry to `Changelog.mdx` with version and date
2. Update `package.json` version field to match
3. Commit with message: `chore(release): vX.Y.Z`
4. Create and push tag: `git tag vX.Y.Z && git push --tags`
